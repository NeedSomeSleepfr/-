## 1. Цель работы

В этой лабораторной я научилacь строить иерархию классов на основе уже существующего класса из ЛР-1. Главная идея — не переписывать код заново, а наследовать то, что уже есть, и добавлять только новые свойства. Также я разобралась с полиморфизмом — когда разные объекты (серверы разных типов) отвечают по-разному на один и тот же вызов (например, на задачу обработки).

## 2. Описание реализованной иерархии классов

### Базовый класс —> Server
Это тот же Server из ЛР-1. Я его не переписывалa, просто взял готовый и добавил динамически два общих метода, которых не было раньше.
Первый метод (get_info) возвращает текстовое описание сервера. Второй метод (process_task) — общая симуляция работы. Базовый сервер выполняет "базовую системную задачу". Главное, что метод называется одинаково для всех наследников — можно вызвать его, не зная заранее, какой именно тип сервера перед тобой.

### Дочерний класс —> DatabaseServer (Сервер Базы Данных)
Наследует всё от базового Server и добавляет своё.
*   **Новые характеристики:** тип базы данных (SQL/NoSQL) и объем диска для хранения данных в ГБ.
*   **Новые возможности:** умеет делать бэкап данных (backup_data()), если сервер находится в активном состоянии.
*   **Изменённое поведение:** переопределен вывод __str__ и get_info, а при вызове process_task этот сервер обрабатывает запросы к базе данных, а не просто базовую задачу.

### Дочерний класс —> WebServer (Веб-сервер)
Наследует всё от базового сервера и добавляет своё.
*   **Новые характеристики:** используемый веб-фреймворк (Django, Apache и т.д.) и флаг поддержки HTTPS (True/False).
*   **Новые возможности:** умеет перезапускать свою службу (restart_service()), что автоматически сбрасывает все текущие подключения до нуля.
*   **Изменённое поведение:** при вызове process_task веб-сервер симулирует отдачу HTML-страниц по нужному протоколу.

### Иерархия
Базовый Server из первой лабораторной стоит на вершине. От него наследуют DatabaseServer и WebServer — они берут весь фундамент (валидацию, статусы, лимиты подключений) и добавляют свои специфичные IT-функции.

## 3. Демонстрация работы

### Сценарий 1 — Создание объектов разных типов
Создаются три типа объектов: обычный Server, две базы данных и два веб-сервера. Для каждого выводится информация и текстовое описание — каждый тип отвечает по-своему, переопределяя родительский текст. Также проверяется принадлежность объектов к типам через isinstance()

![1](../../Image/lab3/03.1.png)

### Сценарий 2 — Уникальные методы каждого класса
Демонстрируются уникальные возможности дочерних классов, которых нет у родителя. База данных делает бэкап, а веб-сервер перезапускает свою службу, обнуляя счетчик подключений. 

![2](../../Image/lab3/03.2.png)

### Сценарий 3 — Полиморфизм без условий
Все пять серверов добавляются в один кластер (ExtendedServerCluster). Затем для каждого сервера в цикле вызывается метод process_task() — без единого условия if. Сервер БД автоматически обрабатывает SQL, веб-сервер отдает страницы, базовый делает свою задачу. Каждый объект сам знает, как себя вести.

![3](../../Image/lab3/03.3.png)

### Сценарий 4 — Состояния и блокировки
Демонстрируется, что наследование сохраняет логику состояний из ЛР-1. Сервер БД падает (вызывается crash()), переходит в статус ошибка, и из-за этого его уникальный метод backup_data() блокируется, выбрасывая ошибку. 

![4](../../Image/lab4/03.4.png)

### Сценарий 5 — Фильтрация по типу объекта
Мы обращаемся к нашей общей коллекции серверов и с помощью специальных методов (get_only_databases, get_only_web_servers) создаем новые коллекции, отфильтрованные по типам.

## 4. Вывод

В этой лабораторной я изучилa наследование и полиморфизм.
Наследование позволяет не дублировать код: дочерние классы получают валидацию, свойства и статусы от родителя и просто расширяют функционал.
Полиморфизм позволяет вызвать один и тот же метод (process_task) для разных объектов коллекции и получить разный результат, избегая громоздких конструкций if type == .... 
Также я понялa, что в коллекциях при работе с наследниками важно использовать isinstance() вместо жесткого сравнения через type(), чтобы родительский контейнер пускал внутрь дочерние объекты.
