# 1. Цель работы

В этой лабораторной я познакомилась с абстрактными классами и интерфейсами (ABC). Объясню своими словами, что это такое.

### Абстрактный класс
Это как шаблон с обязательными требованиями. Сам по себе он ничего не делает и создать объект из него напрямую нельзя. Он просто говорит: "Если хочешь быть частью системы — реализуй вот это". Это как франшиза ресторана — каждый повар обязан уметь готовить, убирать и принимать заказы. Как именно он это делает — его дело. Но эти три вещи должны быть у всех.

### Интерфейс
Это чистый список обязательств и контрактов поведения. Как договор при устройстве на работу сисадмином — подписал, значит обязан предоставлять отчетность. 

# 2. Какие интерфейсы я создала

Для моей IT-инфраструктуры я создала два интерфейса:

### Первый —> IMonitorable (Мониторируемый)
Любой сервер, который реализует этот интерфейс, обязан иметь метод get_metrics(), чтобы отдавать данные для мониторинга. Это очень удобно — можно собрать все серверы в один список и запросить метрики разом, не проверяя, база данных это или обычный прокси.

### Второй —> IComparable (Сравниваемый)
Любой сервер с этим интерфейсом обязан уметь сравнивать себя с другим сервером через метод compare(). При этом каждый делает это по-своему: обычный сервер сравнивает себя по приоритету, база данных — по объему жесткого диска, а веб-сервер — по количеству активных подключений.

# 3. Как классы реализуют интерфейсы

Все три типа моих серверов — обычный Server, DatabaseServer и WebServer — реализуют оба интерфейса (множественное наследование). Метод метрик и метод сравнения есть у каждого, но работают они по-разному.

*   Обычный сервер показывает свои общие настройки и приоритет.
*   База данных выводит в метриках тип SQL/NoSQL и занятое место, а при сравнении оценивает, кто тяжелее по гигабайтам.
*   Веб-сервер показывает используемый фреймворк и статус HTTPS, а при сравнении выясняет, на ком сейчас больше пользователей (подключений).

Создаются три сервера разных типов. Для каждого вызывается метод запроса метрик — одно и то же название метода, но совершенно разный результат. Также проверяется, что каждый объект реально принадлежит к интерфейсу IMonitorable через функцию isinstance().

### Сценарий 2 — Интерфейс "Сравниваемый"
Создаются дополнительные серверы, и каждый сравнивается с сервером своего типа. База данных сравнивается с базой (по ГБ), веб-сервер с веб-сервером (по подключениям).

### Сценарий 3 — Универсальная функция через интерфейс
Создаётся обычная функция print_system_status, которая принимает любой список (даже не коллекцию). Ей всё равно, что там внутри — база, прокси или балансировщик. Она проверяет интерфейс и просто вызывает метод. Каждый объект сам знает, что ответить.

### Сценарий 4 — Фильтрация коллекции по интерфейсу
В наш кластер добавляются серверы разных типов. Вызывается полиморфный метод коллекции, который печатает метрики всех сразу. Затем коллекция фильтруется — создаются новые кластеры только из тех серверов, кто поддерживает нужный интерфейс.

### Сценарий 5 — Сортировка через интерфейс "Сравниваемый"
Создаётся кластер из обычных серверов в случайном порядке. Список выводится на экран, затем коллекция запускает пузырьковую сортировку. Фокус в том, что внутри сортировки используется абстрактный метод compare(). В итоге серверы выстраиваются по возрастанию приоритета.

# 5. Вывод

Раньше казалось, что интерфейсы — это лишняя писанина кода. Но смысл в том, что они дают гарантию. Ты точно знаешь: если объект реализует интерфейс мониторинга, метод получения метрик у него 100% есть. Не нужно проверять конкретные типы дочерних классов, не нужно писать длинные ветвления if/else. Просто вызываешь нужную функцию, и каждый объект сам разбирается, как её выполнить. Архитектура становится предсказуемой.
